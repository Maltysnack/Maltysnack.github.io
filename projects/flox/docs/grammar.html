01 Lexical structure

Source is plain ASCII. Whitespace separates tokens but is otherwise insignificant. Two comment forms are supported:

  • // line comment runs to end of line.
  • /* block comment */ may span multiple lines; does not nest.

Identifiers and numbers

IDENTIFIER → [a-zA-Z][a-zA-Z0-9_]* NUMBER → [0-9]+ ( "." [0-9]+ )?

Identifiers name systems, rivers, dams, outlets, generic nodes, and modes. Case-sensitive. Numbers are decimal literals with an optional fractional part. A leading - on an attribute value (e.g. supply: -40) negates the number.

03 Statements

FormMeaning
mode name;Selects the simulator. river, markov, or grid. Defaults to river if absent. One mode per file; later mode declarations overwrite the file's mode.
system Name;Declares a named system. Subsequent declarations and chains attach to the most recently declared system.
river Name { ... }Declares a river node. River-mode attributes (see Node declarations).
dam Name { ... }Declares a dam node. River-mode only in practice.
outlet Name;Declares a zero-flow accumulator. Block is optional and must be empty.
node Name;Generic state container. Mode-agnostic; takes only optional supply (used by grid mode).
A -> B -> C;Chain of bare references. Nodes must be declared somewhere in the same system.
start Name mass;Markov mode: sets initial mass on the named node. Multiple start statements for the same node accumulate.
list Name;Lists every node in the named system with its outgoing connections.
map Name;Prints the mode-appropriate network diagram.
print Name;Runs the simulator with the mode's default argument and prints the result.
print Name n;River: simulate at n mm rainfall on day 1. Markov / grid: n = tick budget.
print Name [r1, r2, ...];River only: per-day rainfall series in millimetres.
help;Prints a short CLI syntax reference.

The interpreter runs in two passes. Pass 1 collects declarations and chains. A resolve step validates each chain reference, then a mode-specific validator runs. Pass 2 executes list / map / print in source order.

04 Node declarations

River

river Name { base: N, peak: N, decay: N, supply: N }
AttributeRequiredMeaning
baseyesBaseline flow in litres per second with no rainfall.
peakyesMultiplier applied to rainfall (mm) to compute the peak surge.
decaynoFraction of the surge retained each subsequent day. Range 0 to 1. Default 0.7.
supplynoGrid-mode supply (+) or load (-). Ignored in river / markov modes. Default 0.

Dam

dam Name { capacity: N, threshold: N%, normalRate: N, openRate: N }
AttributeRequiredMeaning
capacityyesMaximum storage in megalitres.
thresholdyesStorage level at which the dam opens. Written as a percent (80%) or fraction (0.8).
normalRateyesOutflow as a fraction of inflow when below threshold.
openRateyesOutflow as a fraction of inflow when at or above threshold.

Outlet

outlet Name;

No attributes. Accumulates everything that flows into it. May have outgoing connections (acts as a zero-flow passthrough in that case).

Node (generic)

node Name; node Name { supply: N }
AttributeRequiredMeaning
supplynoGrid-mode supply (+) or load (-). Used only by grid mode. Default 0.

Prefer node in markov and grid programs; river / dam / outlet are river-mode-specific.

05 Chains and arrows

A chain is a sequence of identifiers separated by arrows. There are four arrow forms:

FormMeaning
A -> BDirected edge from A to B with weight 1.
A <- BDirected edge from B to A with weight 1.
A -[w]-> BDirected edge from A to B with weight w.
A <-[w]- BDirected edge from B to A with weight w.

Direction can flip mid-chain. The arrangement is purely a readability choice; each adjacent pair compiles to one directed edge with one weight. Examples:

A -> B -> C; // three-node line C <-[0.5]- A -[0.5]-> B; // A splits half to C, half to B Spring -> Confluence <- Tributary; // two sources, one sink

Grid mode ignores arrow direction (every edge is bidirectional). All four arrow forms still parse but behave the same.

06 Modes

A program selects its mode with a mode declaration at any point in the file. The default is river.

07 River semantics

River hydrograph

Given r millimetres of rainfall on day 0, river flow on day n is:

day 1 (rising): base + 0.3 × peak × r day 2 (peak): base + peak × r day n ≥ 3: base + peak × r × decay^(n - 2)

With a rainfall series, each non-zero day kicks off its own hydrograph; flow on day n is the superposition of every event so far. A series shorter than ten days is padded with zeros.

Dam outflow

If current storage is below threshold, outflow = inflow × normalRate. Otherwise outflow = inflow × openRate. Storage is updated each day by inflow − outflow with units converted at the boundary (L/s → ML/day).

Edge weights

Edge weights scale flow on that edge: flow_on_edge = upstream_flow × weight. Default weight 1. Multiple outgoing edges with default weight all receive 100% of their source (a clone, not a split); use explicit weights to model real splits.

08 Markov semantics

Each node carries a non-negative mass. The initial distribution is set with one or more start statements. Each tick, every node's mass is redistributed across its outgoing edges in proportion to weights, then collected at the targets.

Validation

  • For each node with outgoing edges, the weights must sum to 1.0 (within 1e-6).
  • At least one node must have non-zero initial mass.
  • Every start reference must name a declared node.

Convergence

Simulation halts at the tick budget or when the maximum per-node mass change in a single tick drops below 1e-9. The output labels the result as "converged" or "not yet converged".

Output

map prints the chain as states + transitions grouped by source. print runs the simulator and prints the final distribution.

09 Grid semantics

Each node carries a level (initialised to 0) and an optional fixed supply. Each undirected edge with conductance c between A and B carries flow (levelA − levelB) × c. Each tick:

delta[i] = supply[i] + sum_over_neighbors_j((level[j] - level[i]) × c[ij]) level[i] += 0.1 × delta[i]

The damping factor 0.1 keeps the iteration stable for moderately conditioned networks. Convergence is detected when the largest per-node change falls below 1e-7.

Validation

  • Total supply must sum to zero across all nodes (within 1e-6). Generators (+) and loads (-) balance.
  • At least one node must have non-zero supply.

Bidirectional edges

Arrow direction is ignored: every edge is undirected. Parallel edges between the same pair of nodes sum their conductances. Weighted-arrow notation specifies conductance.

Output

map classifies nodes as generator / load / relay and lists undirected edges with conductances. print shows final per-node levels and signed per-edge flows (positive = first-to-second direction).
